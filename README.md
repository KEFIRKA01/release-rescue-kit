## Витрина

Скриншоты и GIF складываются в `assets/`.

`Набор стабилизации релизов` показывает production-thinking подход: быстро проверить, что сломалось после релиза, собрать картину проблем, оценить риск и дать команде понятный маршрут, что чинить в первую очередь.

## Что показывает проект

- технический rescue-подход для проблемных релизов;
- автоматические проверки доступности и ключевых пользовательских сценариев;
- структурированный отчёт по симптомам и сбоям;
- работу с набором целей из внешнего seed-файла;
- оценку риска сервисного окна и влияния на владельца процесса;
- архитектуру для поддержки, стабилизации и сопровождения действующих веб-проектов.

## Для каких задач подходит

- аварийное восстановление после релиза;
- технический аудит веб-проекта после запуска;
- стабилизация интеграций и критичных сценариев;
- сопровождение старых сервисов с накопленным техдолгом;
- внутренние инструменты release readiness и smoke-check.

## Ключевые сценарии

- быстрая проверка сервиса сразу после выката;
- выявление упавших или деградировавших точек;
- загрузка внешнего списка критичных маршрутов;
- сбор сводного отчёта по релизу;
- запуск аварийного playbook для нестандартного сбоя.

```bash

python -m kit.cli --seed seed/demo_targets.json
```

## Почему это сильный кейс

- показывает редкое для портфолио направление: `rescue + support + stabilization`;
- помогает продавать заказы не только на разработку нового, но и на спасение уже работающего;
- хорошо раскрывает инженерное мышление вокруг релизов, риска и эксплуатационной устойчивости.
